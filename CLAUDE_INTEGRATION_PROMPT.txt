PROJECT NAME: Cricket Auction System
BACKEND TYPE: Spring Boot REST API
BASE URL: http://localhost:8080
FRONTEND: Angular (TypeScript)
AUTHENTICATION: JWT Bearer Token
DATABASE: PostgreSQL

KEY FEATURES:
- Tournament Management (create, update, delete tournaments)
- Team Management (create, update, delete teams with logos)
- Player Registration & Management (public self-registration + admin approval workflow)
- Auction Player Pool (promotion of approved players to auction)
- Live Auction (sell/unsold player tracking)
- Bid Increment Rules (configurable bidding increments)
- Owner Dashboard (comprehensive tournament analytics)

1. AUTHENTICATION ENDPOINTS

1.1 Register New Organizer Account
URL: /api/auth/register
METHOD: POST
AUTH: Not Required (Public)
CONTENT-TYPE: application/json

REQUEST:
{
  "name": "string",                           // Required, 2+ characters, letters/spaces/dots/hyphens only
  "email": "string",                          // Required, valid email format
  "password": "string",                       // Required, 8-100 chars, min 1 uppercase, 1 digit, 1 special char
  "phoneCountryCode": "string",               // Required (e.g., "+91", "+1", "+44")
  "phoneNumber": "string",                    // Required, 6-15 digits
  "organisation": "string",                   // Optional
  "sport": "string",                          // Required (e.g., "CRICKET")
  "numberOfTeams": "integer"                  // Optional
}

RESPONSE (201):
{
  "id": 1,
  "name": "John Organizer",
  "email": "john@example.com",
  "role": "ADMIN"
}

ERROR (400):
{
  "errors": {
    "email": "Email is already in use",
    "password": "must match pattern"
  }
}

   Each service should:
   - Have typed methods for each API endpoint
   - Return Observable<T> for async operations
   - Handle API errors gracefully
   - Use strong typing with TypeScript interfaces

2. HTTP INTERCEPTOR
   - Create an HttpInterceptor to automatically add JWT token to requests
   - Handle 401 responses by clearing token and redirecting to login
   - Add Content-Type headers appropriately

3. TYPESCRIPT INTERFACES/MODELS
   - Create interfaces for all request/response types
   - Create interfaces for all domain models (Tournament, Team, Player, etc.)
   - Use strong typing throughout

4. ENVIRONMENT CONFIGURATION
   - Create environment.ts and environment.prod.ts
   - Store API_BASE_URL in environment
   - Store other config constants

5. GUARDS
   - Create AuthGuard to protect routes requiring authentication
   - Implement canActivate() method

6. STATE MANAGEMENT (optional)
   - Suggest NgRx or similar state management patterns if needed

REQUIREMENTS:
- Use Angular best practices
- Implement error handling consistently
- Include proper TypeScript typing
- Add comments and documentation
- Handle multipart/form-data correctly
- Support JWT token refresh if needed
- Include proper logging for debugging

SPECIFIC IMPLEMENTATION DETAILS:

For Multipart Form Data (Tournaments, Teams, Players, Auction Players):
- Do NOT manually set Content-Type header
- Let the browser set it automatically with boundary
- Use FormData API to build the request

For Image Endpoints:
- Tournament logo: /api/tournaments/{id}/logo
- Team logo: /api/teams/{id}/logo
- Player photo: /api/players/{id}/photo
- Auction player photo: /api/auction-players/{id}/photo
- Use these URLs directly in img src attribute or as blob for download

For Authentication Flow:
1. User calls login() -> receive token
2. Store token in localStorage/sessionStorage
3. Interceptor adds token to all subsequent requests
4. If 401 response, clear token and redirect to login

For Player Workflow:
1. Public registration creates PENDING player
2. Admin approves/rejects PENDING players
3. Approved players can be promoted to AuctionPlayer
4. AuctionPlayers go through auction (AVAILABLE -> SOLD/UNSOLD)

For Auction Players Status:
- AVAILABLE: Ready for bidding
- SOLD: Sold to a team
- UNSOLD: Not sold in auction
- Can requeue UNSOLD back to AVAILABLE

PROVIDE:
1. Complete service files with all methods
2. HTTP interceptor for JWT handling
3. All TypeScript interfaces/models
4. Auth guard
5. Environment configuration files
6. Usage examples for key operations
